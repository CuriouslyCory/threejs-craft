## Goal / scope
Add the server persistence layer for edit-deltas: Prisma models (`SavedWorld`, `ChunkDelta`), a
pure `active-world` helper over an injected `db`, and a rewritten `worldRouter` exposing
`protectedProcedure`s (`load` / `applyEdit{worldId,command}` / `newGame` / `status`). Implements
ADR-0001's storage over the wire/DB boundary. **The game still reads the literal seed — no
runtime/client change yet** (that's #20). Verified via router + helper unit tests.

Out of scope: client wiring / persist queue / auth-aware load (#20); pause menu (#21); inventory
and player-position persistence (never — ADR-0001 covers block edit-deltas only).

## Deliverables

### 1. Prisma models — `prisma/schema.prisma` (match tab style)
- `SavedWorld { id String @id @default(cuid()); userId String @unique; seed String;
  version Int @default(1); updatedAt DateTime @updatedAt; user User @relation(fields:[userId],
  references:[id], onDelete: Cascade); deltas ChunkDelta[] }` — exactly **one active world per user**.
- `ChunkDelta { savedWorldId String; chunkKey String; data Bytes; savedWorld SavedWorld @relation(
  fields:[savedWorldId], references:[id], onDelete: Cascade); @@id([savedWorldId, chunkKey]) }` — the
  composite PK enforces one row per chunk (the single-record-per-key precondition #18's bridge relies on).
- Add `savedWorld SavedWorld?` back-relation to `User`.
- Run `pnpm db:generate` (regenerates the Prisma client). Commit the schema; migrations per repo
  convention (check whether the repo commits `migrations/` or uses `db push` — inspect `package.json`
  `db:*` scripts and existing `prisma/` layout before choosing; do NOT invent a migration flow).

### 2. Active-world helper — `src/server/world/active-world.ts` (pure over injected `db`)
Pure logic taking an injected `db` (unit-tests with a fake db, no Prisma). Each mint-seed-injectable
(`mintSeed = () => crypto.randomUUID()` default — real server entropy, keeps worldgen pure).
- `loadWorld(db, userId, mintSeed?)` → find the user's `SavedWorld`; if none, **auto-provision** one
  with a fresh minted seed (empty deltas); return `{ worldId, seed, version, deltas: ChunkDeltaRecord[] }`.
  **Never re-mints a seed for an existing world.**
- `recordEdit(db, {userId, worldId, command})` inside a `$transaction`:
  1. load the world (assert it belongs to `userId` and `worldId` matches — reject stale worldId);
  2. `hydrateWorld(seed, storedDeltas)`; apply the command's **voxel mutation** to the reconstructed
     world (see Decision D1); 3. for each `changed` chunkKey, `computeChunkDelta(seed, chunkKey, snapshot)`;
  4. if `isEmptyChunkDelta` → **delete** the `ChunkDelta` row (revert-to-base prune); else **upsert** it;
  5. bump `updatedAt`/version. Two edits to the same chunk must **accumulate** (recompute from stored
     state each time — lost-update guard).
- `newGame(db, userId, mintSeed?)` = `$transaction`: **delete** the user's `SavedWorld` (cascade purges
  deltas) then **insert** a fresh-seed row → a new seed always lands on an empty row (ADR-0003's
  atomic purge-then-reseed; "never write a new seed over surviving deltas" holds by construction).
- `status(db, userId)` → cheap `{ worldId, seed, version, updatedAt }` (Save's confirm target).

### 3. Rewrite `worldRouter` — `src/server/api/routers/world.ts` (all `protectedProcedure`)
Replace the stub entirely. Base64 boundary lives HERE (`Buffer` is fine — server-only file):
- `load` (query): `loadWorld(ctx.db, ctx.session.user.id)`; return `{ worldId, seed, version,
  deltas: [{ chunkKey, data: <base64 string> }] }` — `data.toString("base64")` on read.
- `applyEdit` (mutation): input `z.object({ worldId: z.string(), command: CommandSchema })`. The
  `worldId` envelope rejects edits queued before a New Game. Calls `recordEdit`. On the DB write side,
  `Buffer.from(<base64>, "base64")` — but note deltas are computed as `Uint8Array` internally; convert
  to `Buffer` only at the Prisma `data` column.
- `newGame` (mutation): `newGame(ctx.db, ...)`; return the fresh `{ worldId, seed, version }`.
- `status` (query): cheap metadata query.
- Signed-out → `protectedProcedure` throws `UNAUTHORIZED` (no explicit check needed).

### 4. ADRs
- `docs/adr/0002-per-user-world-persistence.md` — per-user world persistence, **single active save**
  (`SavedWorld.userId @unique`). Record the OPEN alternative for reviewer sign-off: a multiplayer-

## Decisions to record & surface at PR review
- **D1 (implementation decision, flag for reviewer):** the server applies only the command's **voxel
  mutation** (break ⇒ Air at `command.at`; place ⇒ `command.block` at `command.at`) to the
  reconstructed chunk — it does **NOT** re-run reach/inventory gating, because inventory and
  player-position are not persisted server-side (out of scope). The authenticated client is trusted
  for gameplay validity; the server owns persistence integrity only. Alternative (full server-side
  `applyCommand` re-validation) would require persisting inventory/player state — explicitly rejected.
- **ADR-0002** (single active world/user) and **ADR-0003** (fresh-seed purge-then-reseed).
- **OPEN:** `SavedWorld.userId @unique` vs `World { ownerId }` — reviewer accepts or requests.
