# Plan: #14 — Deepen the world-state seam into one `WorldStore` module

**Issue:** #14 Deepen the world-state seam: one WorldStore module (fixes blocks placed into new chunks never rendering)
**Branch:** `feat/architecture-deepening` (implement on a topic branch off it, e.g. `feat/14-world-store-seam`)
**Type:** Deep-module consolidation + one user-visible bug fix (new-chunk render). Behavior change is **exactly one**: placing a block into a chunk created after mount now renders immediately. Everything else is behavior-preserving.

---

## Objective & non-goals

**Objective.** Merge the three shallow world-state modules — `GameStore` (mutation + versioning), `WorldSource`/`LocalWorldSource`/`createLocalWorldStore` (read surface), and the identity pass-through `RemoteWorldSource` — into **one deep `WorldStore` module** whose interface is the seam a future remote adapter (per ADR-0001) will actually stand behind:

1. `apply(command) → CommandResult` (mutation), shared by consuming code and any future remote store.
2. **Versioned chunk entries that carry their `ChunkKey`** alongside `origin` and `chunk` — identity computed once inside the store, never reconstructed downstream via `coord → origin → coord → key`.
3. `subscribe` + a single `getSnapshot` for `useSyncExternalStore`, where **chunk existence is part of the versioned snapshot** — so a chunk first created by an edit mounts a mesh immediately.

**The bug this fixes.** Today the scene snapshots the set of chunks **once at mount** (`game-scene.tsx:120`, a `useMemo(..., [store])`). Placing a block into a chunk that did not exist at mount (e.g. building a tower up past generated terrain into the next chunk layer) mutates `World` and bumps the version counter, but the newly-created chunk never enters the mounted entry list, so no `ChunkMesh` is ever mounted for it — the placed block is invisible.

**Non-goals.**
- No change to the frozen `Command`/`CommandResult` Zod contract (`src/game/command.ts`, `src/game/command-schema.ts`) — see the ADR-0001 checkpoint below. Byte-for-byte untouched.
- No real networking / no real remote adapter. The identity `RemoteWorldSource` is **deleted**, not replaced; the seam it marked becomes the `WorldStore` interface itself.
- No change to `applyCommand`, `canBreak`, `canPlace`, `World`, `Chunk`, `coords`, `Inventory`, worldgen, the renderer math (`chunk-instances.ts`, `chunk-mesh.tsx` internals), or the player/collision code.
- No change to inventory / hotbar semantics (`selectSlot`/`cycleSelection`/`getInventorySnapshot` carry over verbatim).

All symbol names below are grep-verified against the branch. The only **new** names introduced are the consolidation targets: `WorldStore`, `createWorldStore`, `WorldChunkEntry` (and `ChunkOrigin`, relocated verbatim from `local-world-store.ts`).

### The bug's exact location (chunk existence snapshotted once at mount)

`src/app/game/game-scene.tsx`:

- **`:120`** — `const chunkEntries = useMemo(() => store.source.chunkEntries(), [store]);`
  `store` is `createLocalWorldStore(...)` memoized with `[]` at `:93-96`, so its identity **never changes** for the component's lifetime. Therefore `chunkEntries` is computed **exactly once, at mount**, and captures only the chunks that existed then.
- **`:112`** — `useSyncExternalStore(gameStore.subscribe, gameStore.getVersionSnapshot);` re-renders the scene on every successful `apply` (a monotonic number bump), **but that re-render re-runs the `:120` memo with unchanged deps → the same stale entry list.**
- **`:172-184`** — the render maps `chunkEntries` to `<ChunkMesh>` and, per entry, reconstructs the key it just threw away:
  ```
  {chunkEntries.map(({ chunk, origin }) => {
    const { cx, cy, cz } = worldToChunkCoord(origin.x, origin.y, origin.z); // coord ← origin
    const key = chunkKey(cx, cy, cz);                                        // key ← coord
    return <ChunkMesh key={key} chunk={chunk} origin={origin} atlas={atlas}
             version={gameStore.getChunkVersion(key)} />;
  })}
  ```
  This is the `coord → origin` (in `LocalWorldSource.chunkEntries`) → `origin → coord → key` (here) round-trip the issue calls out.

**Mechanism, end to end.** `apply` → `applyCommand` → `World.setBlock` → `World.ensureChunk(cx,cy,cz)` (`world.ts:48`) creates a brand-new `Chunk` and inserts it into `World.chunks`. `apply` bumps `chunkVersions` for the new key and the monotonic `version`, and calls `notify()`. The scene re-renders (via `:112`) — but `chunkEntries` (`:120`) is frozen at mount, so the new chunk is absent from the mapped list; **no `ChunkMesh` is ever mounted for it.** Editing an *existing* chunk works only because that chunk was already in the mount-time list, so its already-mounted `ChunkMesh` simply gets a new `version` prop (`:181`) and recomputes.

## Target design

### One module: `src/game/store/world-store.ts`

Reuse the existing path (`world-store.ts` already imports `applyCommand`, `Inventory`, `ChunkKey`). Rename `GameStore → WorldStore`, absorb the read surface, and make chunk entries versioned + key-carrying.

**New exported types** (relocate `ChunkOrigin` verbatim from `local-world-store.ts`; add `WorldChunkEntry`):

```ts
/** World-space coordinate of a chunk's local (0,0,0) voxel. */
export interface ChunkOrigin {
  readonly x: number;
  readonly y: number;
  readonly z: number;
}

/**
 * One loaded chunk in the versioned snapshot: its ChunkKey (identity computed
 * once, here — never reconstructed from `origin` downstream), its world-space
 * origin, the live `Chunk` (mutated in place by `apply`), and its per-chunk
 * version (bumped once per successful `apply` that lists this chunk in
 * `CommandResult.changed`).
 */
export interface WorldChunkEntry {
  readonly key: ChunkKey;
  readonly origin: ChunkOrigin;
  readonly chunk: Chunk;
  readonly version: number;
}
```

**Public interface of `WorldStore`:**

```ts
class WorldStore {
  constructor(world: World, inventory?: Inventory);

  // Mutation — the seam a future RemoteWorldSource stands behind. Unchanged signature.
  apply(command: Command, from: Vec3, reach?: number, playerPosition?: Vec3): CommandResult;

  // Inventory selection (verbatim from GameStore).
  selectSlot(index: number): void;
  cycleSelection(delta: number): void;

  // useSyncExternalStore wiring.
  subscribe(listener: () => void): () => void;

  // THE versioned snapshot: existence + per-chunk version, each entry carrying its key.
  // Identity-stable between renders when nothing changed; a NEW array only after a
  // successful `apply` that added a chunk or bumped a chunk's version.
  getSnapshot(): readonly WorldChunkEntry[];

  // Inventory snapshot for the hotbar (verbatim from GameStore).
  getInventorySnapshot(): Inventory;
}

export function createWorldStore(world: World, inventory?: Inventory): WorldStore;
```

**Removed vs. `GameStore`:** `getChunkVersion(key)` and `getVersionSnapshot()` are gone — per-chunk version now rides on each `WorldChunkEntry.version`, and "did anything change" is expressed by `getSnapshot()`'s array identity. `LocalWorldStore`, `WorldSource`, `LocalWorldSource`, `ChunkSourceEntry`, `createLocalWorldStore` are gone (folded away). `generated` is no longer held by the store — game-scene keeps `GeneratedWorld` directly.

**Snapshot construction (the deep part — preserves the O(dirty-chunks) rebuild invariant).**
The store caches a per-key entry map and a top-level snapshot array. `getSnapshot()` must be identity-stable when nothing changed (React's `useSyncExternalStore` contract — otherwise "getSnapshot should be cached" warnings / render loops). Rebuild happens **only inside `apply` on success**, and preserves per-entry object identity for chunks that neither appeared nor changed version — so `ChunkMesh`'s `useMemo([chunk, origin, version])` still recomputes **only** the dirty/new chunk, never the whole world.

```ts
private readonly chunkVersions = new Map<ChunkKey, number>();
private readonly entryCache = new Map<ChunkKey, WorldChunkEntry>();
private snapshot: readonly WorldChunkEntry[] = this.rebuildSnapshot();

getSnapshot = (): readonly WorldChunkEntry[] => this.snapshot;

/** Rebuild the cached entry list from the live World, reusing the existing
 *  entry object for any chunk whose (chunk instance, version) is unchanged so
 *  its identity — and thus ChunkMesh's memo — survives. */
private rebuildSnapshot(): readonly WorldChunkEntry[] {
  const next: WorldChunkEntry[] = [];
  for (const { cx, cy, cz, chunk } of this.world.chunkEntries()) {
    const key = chunkKey(cx, cy, cz);
    const version = this.chunkVersions.get(key) ?? 0;
    const cached = this.entryCache.get(key);
    if (cached && cached.chunk === chunk && cached.version === version) {
      next.push(cached);
      continue;
    }
    const entry: WorldChunkEntry = {
      key,
      origin: { x: cx * CHUNK_SIZE, y: cy * CHUNK_SIZE, z: cz * CHUNK_SIZE },
      chunk,
      version,
    };
    this.entryCache.set(key, entry);
    next.push(entry);
  }
  return (this.snapshot = next);
}
```

`apply`, after a successful `applyCommand` result, does exactly what `GameStore.apply` did (set inventory, bump `chunkVersions` for each `result.changed` key) and then calls `this.rebuildSnapshot()` before `notify()`. The monotonic `version` counter is dropped (no longer needed — the snapshot array identity is the change signal). `selectSlot`/`cycleSelection` call `notify()` **without** rebuilding the chunk snapshot: an inventory-only change leaves `getSnapshot()` identity-stable, so the scene correctly does not re-render while the hotbar (via `getInventorySnapshot`) does.

> Design note (not an unknown): `getSnapshot` returns the **entries array itself** as the versioned snapshot, rather than an opaque counter the scene then reads entries beside. This is the "chunk existence becomes part of the versioned snapshot" the issue asks for, in one value. It is safe with `useSyncExternalStore` because the array is cached and only replaced on a real change.

### How React consumes it (`src/app/game/game-scene.tsx`)

## New regression test spec (pure, no renderer)

Add to `src/game/__tests__/store/world-store.test.ts` (the acceptance-criteria demoable-fix, expressed as a pure store test — the exact path the bug lives on):

```ts
describe("WorldStore.getSnapshot — new-chunk render path", () => {
  it("a chunk first created by a PlaceBlock edit appears in the versioned snapshot", () => {
    const world = new World();                 // empty: zero loaded chunks
    const inventory = addDrop(createInventory(), BlockType.Stone);
    const store = createWorldStore(world, inventory);

    const at = { x: 1, y: 0, z: 0 };           // chunk (0,0,0) — absent in an empty world
    const key = chunkKey(...Object.values(worldToChunkCoord(at.x, at.y, at.z)) as [number,number,number]);
    // (or compute cx/cy/cz explicitly; key === "0,0,0")

    // Before the edit: the target chunk is not in the snapshot at all.
    expect(store.getSnapshot().some((e) => e.key === key)).toBe(false);

    const before = store.getSnapshot();
    const result = store.apply(
      { type: "PlaceBlock", at, block: BlockType.Stone },
      { x: 0, y: 0, z: 0 },     // from
      undefined,                 // default reach
      { x: 1000, y: 0, z: 1000 } // playerPosition far away → no self-clip
    );

    expect(result.ok).toBe(true);

    // After the edit: the newly-created chunk is now a versioned entry, carrying
    // its own key + world-space origin + a bumped version — the mesh WOULD mount.
    const after = store.getSnapshot();
    expect(after).not.toBe(before);            // snapshot identity changed → scene re-renders
    const entry = after.find((e) => e.key === key);
    expect(entry).toBeDefined();
    expect(entry!.origin).toEqual({ x: 0, y: 0, z: 0 });
    expect(entry!.version).toBe(1);
    expect(entry!.chunk.get(1, 0, 0)).toBe(BlockType.Stone);
  });

  it("editing an existing chunk keeps unchanged chunks' entry identity (O(dirty) rebuild)", () => {
    const world = new World();
    world.setBlock(1, 0, 0, BlockType.Stone);   // chunk A (0,0,0)
    world.setBlock(20, 0, 0, BlockType.Stone);  // chunk B (1,0,0)
    const store = createWorldStore(world);

    const before = store.getSnapshot();
    const keyB = chunkKey(1, 0, 0);
    const entryBefore = before.find((e) => e.key === keyB)!;

    store.apply({ type: "BreakBlock", at: { x: 1, y: 0, z: 0 } }, { x: 1, y: 0, z: 0 });

    const after = store.getSnapshot();
    // Unchanged chunk B kept its exact entry object → ChunkMesh memo survives.
    expect(after.find((e) => e.key === keyB)).toBe(entryBefore);
  });
});
```

Both are pure — no `three`, no DOM, no renderer — exactly like the existing store tests.
